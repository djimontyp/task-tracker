# Entity Hierarchy

> **Status:** Validated with user (2025-12-28)

## Overview

Pulse Radar has a clear 3-layer entity hierarchy:

```
┌─────────────────────────────────────────────────────┐
│                      TOPICS                          │
│        (верхній рівень: Mobile, Frontend, Backend)   │
│  ┌─────────────────────────────────────────────────┐│
│  │                    ATOMS                        ││
│  │    (знання: problems, decisions, insights)      ││
│  │  ┌─────────────────────────────────────────┐   ││
│  │  │              MESSAGES                   │   ││
│  │  │         (raw data, hidden layer)        │   ││
│  │  └─────────────────────────────────────────┘   ││
│  └─────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────┘
```

## Entity IDs (Technical)

> [!important] UUID Standard
> All entity IDs are **string UUIDs** (not numeric).
> Fixed 2025-12-29.

| Entity | ID Type | Example |
|--------|---------|---------|
| Topic | `string` (UUID) | `"a1b2c3d4-e5f6-7890-..."` |
| Atom | `string` (UUID) | `"f1e2d3c4-b5a6-7890-..."` |
| Message | `string` (UUID) | `"1a2b3c4d-5e6f-7890-..."` |

**Frontend files affected:**
- `features/atoms/types/index.ts` — `Atom.id: string`
- `features/atoms/api/atomService.ts` — all methods
- `features/knowledge/api/versioningService.ts`
- `features/knowledge/components/*`

## Topics (Top Level)

**Purpose:** Organizational categories for knowledge grouping.

**Examples:**
- Mobile Development
- Frontend (React)
- Backend (Python)
- DevOps
- AI/ML
- Security

**Properties:**
- `name` — unique identifier
- `icon` — visual representation (Lucide)
- `color` — hex color for UI
- `description` — what this topic covers

**User Interaction:**
- Browse topics to see what areas have activity
- Drill-down into topic to see related atoms
- Topics are auto-generated by AI, user can adjust

## Atoms (Knowledge Units)

**Purpose:** Extracted knowledge units — the core value of the system.

**Types (9):**
| Type | Description | Icon |
|------|-------------|------|
| `problem` | Issue, bug, blocker | AlertTriangle |
| `solution` | Fix, workaround | CheckCircle |
| `decision` | Consensus, approved direction | Scale |
| `question` | Uncertainty, needs clarification | HelpCircle |
| `insight` | Observation, pattern found | Lightbulb |
| `idea` | Proposal, suggestion | Sparkles |
| `blocker` | Something blocking progress | Ban |
| `risk` | Potential future problem | AlertOctagon |
| `requirement` | Must-have, specification | FileText |

**Statuses:**
```
DRAFT → PENDING_REVIEW → APPROVED / REJECTED (archived)
```

**User Interaction:**
- Review pending atoms (approve/reject)
- Browse approved atoms by topic
- Search across all atoms
- Link atoms to each other (relationships)

## Messages (Hidden Layer)

**Purpose:** Raw data from communication channels. Not for daily use.

**Source:** Telegram, Slack, API (future)

**Processing Pipeline:**
```
Message → ImportanceScorer → classification (signal/noise)
       → KnowledgeOrchestrator → Atom extraction
       → Topic assignment
```

**User Interaction:**
- **End User:** Does NOT interact with messages
- **Admin:** Debug access when extraction fails or needs tuning

**Visibility:**
- NOT in main sidebar
- Access via Admin Panel or Settings → Debug
- Like logs — you don't read them until something breaks

## UI Implications

### Sidebar Order (proposed)

```
END USER:
├─ Dashboard (Overview)
├─ Topics (Browse by category)
├─ Atoms (Daily Review)
├─ Search
└─ Settings

ADMIN (additional):
├─ Messages (Debug)
├─ Providers (LLM)
├─ Prompts (Tuning)
└─ System Health
```

### Navigation Patterns

| From | To | Method |
|------|----|--------|
| Dashboard | Topic | Click on topic card |
| Topic | Atoms | Automatic (atoms listed in topic) |
| Atom | Topic | Breadcrumb / back button |
| Atom | Source Message | Admin only, via inspect modal |
