# Частина 1: Простір проблем

## 1.1 Основна проблема

### Головний біль

| Проблема | Опис |
|----------|------|
| Розсіяна комунікація | Багато каналів (Telegram, Jira, тощо) |
| Втрата інформації | Важливе губиться в потоці |
| Відсутність структури | Немає бази знань |
| Пошук рішень | Витрачається час на "що вирішили" |

### Статистика (Forrester 2025)

> **2.3 години/день** — стільки knowledge workers втрачають на пошук інформації
>
> **80%** лідерів вже використовують AI для продуктивності

### Цільові метрики

- **Економія часу** — на review
- **Пульс проектів** — активність, блокери, проблеми
- **База знань** — накопичення та актуалізація

# Частина 2: Бачення продукту

## 2.1 Формулювання бачення

> **Pulse Radar** збирає інформацію з усіх джерел комунікації, фільтрує шум, витягує структуровані знання і підтримує їх в актуальному стані — щоб команда не втрачала важливе і економила час.

## 2.2 Основні цінності (пріоритет)

### 1. Економія часу

| Принцип | Реалізація |
|---------|------------|
| Швидкий апрув | 1-3 хвилини, не години |
| Мінімальне навантаження | Не зайобувати користувача |

### 2. Пульс проектів

| Аспект | Що показує |
|--------|------------|
| Активність | Що відбувається |
| Блокери | Проблеми та перешкоди |
| Фокус | "Що є зараз", не "план vs факт" |

### 3. База знань

| Властивість | Гарантія |
|-------------|----------|
| Фіксація | Все записується |
| Версіонування | Історія змін |
| Пошук | Все можна знайти

## 2.3 Часові рамки

| Період | Фокус |
|--------|-------|
| День | Що відбулось сьогодні |
| Робочий тиждень | Підсумок тижня |
| Місяць | Загальна картина |

## Ключові принципи

1. **Owner = Single Point of Accountability**
   - Всі рішення проходять через Owner
   - Claude виконує, Owner затверджує

2. **CEO/PM як консультанти**
   - Надають feedback на демо
   - Валідують user stories
   - НЕ блокують розробку

3. **Claude Code як виконавець**
   - Пише код під керівництвом Owner
   - Пропонує рішення, Owner вибирає
   - Автоматизує рутину

## Escalation Path

```
Issue → Owner decides → If major impact → Consult CEO/PM → Owner decides
```

**Правило:** Консультація з CEO/PM не означає очікування рішення. Owner приймає рішення сам.

# Частина 6: Типи Atom

## 6.1 Базові типи (9)

### Core типи

| Тип | Опис | Приклад |
|-----|------|---------|
| **DECISION** | Важливе рішення команди | "Вирішили використовувати PostgreSQL" |
| **PROBLEM** | Виявлена проблема/біль | "API повільний при 1K запитів" |
| **SOLUTION** | Рішення проблеми | "Додали кешування Redis" |
| **QUESTION** | Відкрите питання | "Як краще зробити авторизацію?" |

### Knowledge типи

| Тип | Опис | Приклад |
|-----|------|---------|
| **INSIGHT** | Цінний висновок | "80% юзерів на mobile" |
| **IDEA** | Пропозиція для розгляду | "Можна додати dark mode" |

### PM типи

| Тип | Опис | Приклад |
|-----|------|---------|
| **BLOCKER** | Що блокує прогрес | "Чекаємо на дизайн від клієнта" |
| **RISK** | Потенційна проблема | "API може змінитись у v2" |
| **REQUIREMENT** | Вимога до продукту | "Має працювати offline" |

## 6.2 Розширюваність

| Аспект | Правило |
|--------|---------|
| Custom types | Per-project в v2.0+ |
| Невизначений тип | Human review |
| Ліміт | Максимум 20-25 типів |

## 6.3 Статуси Atom

```
DRAFT → PENDING_REVIEW → APPROVED / REJECTED
```

# Частина 8: Граф знань (Zettelkasten)

## 8.1 Принципи

| Принцип | Опис |
|---------|------|
| **Повноцінний граф** | Не ієрархія, а мережа |
| **Atomic notes** | Одна ідея = один atom |
| **Зв'язки** | Головна цінність системи |
| **Backlinks** | Хто посилається на цей atom |

## 8.2 Типи зв'язків

| Зв'язок | Приклад |
|---------|---------|
| `solved_by` | Problem → Solution |
| `caused_by` | Effect → Cause |
| `related_to` | Atom ↔ Atom |
| `extends` | Base → Extension |
| `contradicts` | Thesis ↔ Antithesis |

> **Розширюваність:** Якщо AI не визначає тип — human review. Людина може запропонувати новий тип.

## 8.3 Створення зв'язків

```
AI автоматично (embeddings + context)
        │
        ▼
Human-in-the-loop (approve/edit)
        │
        ▼
Автоматизація (після калібровки)
        │
        ▼
Людина може редагувати завжди
```

| Етап | Що відбувається |
|:----:|-----------------|
| 1 | AI автоматично (embeddings + context) |
| 2 | Human-in-the-loop (approve/edit) |
| 3 | Автоматизація (після калібровки) |
| — | Людина може редагувати завжди |

## 8.4 Orphans (непов'язані atoms)

| Підхід | Дія |
|--------|-----|
| Показувати | Як потенційну проблему |
| Suggestions | Пропонувати зв'язування |
| Drop | Якщо виявився шумом |

## 8.5 Transclusion

> Atom може включати (embed) інший atom — не тільки посилання, а вбудований контент.

## 8.6 Візуалізація

| Компонент | Пріоритет |
|-----------|:---------:|
| Списки + фільтри | **P1** |
| Backlinks sidebar | **P1** |
| Graph view | P3 |

| Фаза | Опис | Тривалість |

| 2 | Калібровка — правки промптів | 1-4 тижні |
| 3 | Auto + Review — high confidence = auto | Після калібровки |

# Частина 11: User Stories

## Формат історій

> **Як** [роль], **я хочу** [ціль/бажання] **щоб** [вигода/причина].

| Пріоритет | Значення |
|:---------:|----------|
| **Must** | Обов'язково для MVP |
| **Should** | Важливо, але не критично |

## Epic 1: Щоденний Review (PM)

### US-001: Огляд Dashboard [Must] [M]

**Як** PM,
**я хочу** бачити dashboard з ключовими активностями за сьогодні
**щоб** розуміти статус проекту за 5 хвилин замість 30.

**Estimate:** M (1-2 дні) — частково реалізовано, потребує polish

**Критерії прийняття:**
- Dashboard показує кількість повідомлень за сьогодні
- Dashboard показує нові витягнуті Atoms
- Dashboard показує активні Topics
- Dashboard завантажується за <2 секунди

### US-002: Фільтр Signal/Noise [Must] [M]

**Як** PM,
**я хочу** бачити тільки важливі повідомлення (signals)
**щоб** не витрачати час на шум.

**Estimate:** M (1-2 дні) — UI toggle + фільтрація, backend готовий

**Критерії прийняття:**
- Повідомлення класифікуються як Signal або Noise
- За замовчуванням показуються тільки Signals
- Можу перемикатись щоб бачити всі повідомлення
- Видно співвідношення Signal/Noise

### US-003: Atoms за сьогодні [Must] [M]

**Як** PM,
**я хочу** бачити Atoms витягнуті сьогодні
**щоб** швидко зрозуміти які рішення, завдання, проблеми з'явились.

**Estimate:** M (1-2 дні) — AtomsPage існує, потрібен фільтр + approve/reject

**Критерії прийняття:**
- Atoms згруповані за типом (DECISION, PROBLEM, тощо)
- Кожен Atom показує source message
- Можу затвердити/відхилити Atoms
- Видно кількість Atoms по типам

### US-004: Навігація по Topics [Should]

**Як** PM,
**я хочу** переглядати знання по Topics
**щоб** фокусуватись на конкретних областях проекту.

**Критерії прийняття:**
- Список Topics з іконками та кольорами
- Клік на topic - бачу пов'язані Atoms
- Topic показує кількість повідомлень
- Можу створювати/редагувати Topics

- Можна експортувати як звіт

**Як** CTO,
**я хочу** бачити активність по всіх проектах
**щоб** ідентифікувати патерни та вузькі місця.

**Критерії прийняття:**
- Вигляд показує всі проекти
- Метрики активності по кожному проекту
- Доступний порівняльний вигляд
- Drill-down до деталей проекту

## Epic 3: Пошук знань (Developer)

### US-020: Пошук за ключовими словами [Must] [M]

**Як** Developer,
**я хочу** шукати повідомлення та atoms за ключовими словами
**щоб** швидко знаходити релевантну інформацію.

**Estimate:** M (1-2 дні) — search input в header + results page

**Критерії прийняття:**

- Результати підсвічені ключовим словом

### US-021: Семантичний пошук [Won't v1.0]

**Як** Developer,
**я хочу** шукати за змістом, а не тільки за ключовими словами
**щоб** знаходити пов'язану інформацію навіть з іншим формулюванням.

**Відкладено до:** v1.2+

### US-022: Контекст рішення [Should]

**Як** Developer,
**я хочу** бачити чому було прийнято рішення
**щоб** розуміти контекст перед внесенням змін.

**Критерії прийняття:**
- DECISION atom показує source messages
- Часова шкала пов'язаних обговорень
- Учасники рішення

Перед тим як Story йде в Sprint:

### Обов'язкові (10)

| # | Критерій | Опис |
|:-:|----------|------|
| 1 | User Story format | As/I want/So that |
| 2 | Acceptance Criteria | Мінімум 3 критерії |
| 3 | UI mockup/wireframe | Готовий для UI stories |
| 4 | API contract synced | `just api-sync` виконано |
| 5 | Dependencies | Ідентифіковані |
| 6 | No blockers | Немає блокуючих залежностей |
| 7 | Estimate | T-shirt size (S/M/L/XL) |
| 8 | Test cases outline | Основні сценарії описані |
| 9 | Success metrics | Як виміряти успіх |
| 10 | Edge cases | Граничні випадки задокументовані |

### UI Components (додатково)

| # | Критерій | Опис |
|:-:|----------|------|
| 11 | Storybook story | Prepared ДО розробки |
| 12 | Design System | Compliance перевірено |

### Metadata

| # | Критерій | Опис |
|:-:|----------|------|
| 13 | Browser support | Визначено (Chrome, Firefox, Safari) |
| 14 | Related stories | Залінковані |

|:----:|------|---------|

| Output | Опис |
|--------|------|
| **Atoms** | Структуровані знання |
| **Зв'язки** | Типізовані links між atoms |
| **Batch** | Прив'язка до batch |
| **Source mapping** | Повідомлення → atom |

## 18.3 RAG контекст

| Параметр | Значення |
|----------|----------|
| Використовувати | **Так** |
| Вплив | Великий для довготривалої роботи |
| Обережність | Не шкодити якості через старий контекст |

**Включає:**
- Попередні atoms
- Глосарій проекту
- Опис проекту

## 18.4 Калібровка

| Фаза | Тривалість | Що відбувається |
|:----:|:----------:|-----------------|
| Initial | 1-2 тижні | Human review все |
| Tuning | 2-4 тижні | Правки промптів, thresholds |
| Auto | Після | High confidence = auto-approve |

# Частина 24: Framework самоперевірки

> **Мета:** Не "загонятись" в технічні деталі, тримати фокус на продукті.

## 24.1 Чеклист Product-First

> **Перед кожною фічею запитай:**

| Питання | Деталі |
|---------|--------|
| Хто користуватиметься? | Конкретна людина, не "система" |
| Яку проблему вирішує? | Біль користувача |
| Як дізнається що є? | Discovery |
| Як виглядає успіх? | Метрика, не просто "працює" |
| Що якщо не робити? | Реальний пріоритет |

## 24.2 Попередження про Backend Bias

| Ознака | Що робити |
|--------|-----------|
| "Треба зробити API для..." | Стоп! Хто викличе цей API? Навіщо? |
| "Архітектурно правильно..." | А користувачу це дасть що? |
| "Потім знадобиться..." | YAGNI! Чи точно потім? |
| "Складна логіка на беку" | Чи можна спростити UX замість? |
| "Оптимізація..." | Чи є реальна проблема з performance? |
| "Рефакторинг..." | Чи блокує це користувача? |

## 24.3 Щотижнева самоперевірка (5 хвилин)

> **Щоп'ятниці запитай себе:**

| # | Питання |
|:-:|---------|
| 1 | На що я витратив найбільше часу цього тижня? |
| 2 | Це наближає до цінності для CEO/PM? |
| 3 | Що я відклав? Чому? |
| 4 | Що б сказав користувач якби побачив результат? |
| 5 | Чи я "загнався" в технічну нору? |
| 6 | Що ОДНЕ найважливіше на наступний тиждень? |

## 24.4 Тест запитання CEO

Перед плануванням фічі уяви що CEO питає:

> "Окей, і що? Як це допоможе мені бачити пульс проекту за 30 секунд?"

**Якщо не можеш відповісти - можливо це не пріоритет.**

## 24.5 Формула пріоритету

> **Пріоритет** = (Цінність для користувача × Частота використання) / Складність

| Рівень | Опис |
|:------:|------|
| **Високий** | Щоденне використання, вирішує головний біль |
| **Середній** | Корисно, але не критично |
| **Низький** | "Nice to have", рідко використовується |
| **Відкласти** | Технічно цікаво, але користувач не помітить |

## 24.6 Червоні прапорці (СТОП і подумай)

| Прапорець | Опис |
|-----------|------|
| Тиждень без результату | Працюю над фічею без видимого результату |
| Не можу пояснити | Не можу простими словами описати що роблю |
| "На всяк випадок" | Додаю функціонал без чіткої потреби |
| Передчасна оптимізація | Оптимізую те, що ще не працює |
| Рефакторинг замість цінності | Рефакторю замість доставки фіч |
| 2 тижні без показу | Не показував результат нікому |

## 24.7 Ролі підтримки

| Роль | Що дає | Як використати |
|------|--------|----------------|
| **CEO/PM як тестер** | Реальний фідбек | Показувати раз на тиждень |
| **Rubber Duck** | Пояснити = зрозуміти | Розказати вголос план |
| **PRD як компас** | Не забути бачення | Звірятись перед спринтом |
