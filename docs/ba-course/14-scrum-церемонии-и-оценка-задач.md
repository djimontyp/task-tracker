# Техники оценки задач

# Оценка

это прогноз, построенный на основе имеющейся информации.

# Оценка по аналогии

Новая задача займет примерно столько же, сколько похожая старая.

Точность повышается, при работе с похожими проектами в неизменном контексте (методология, перечень заинтересованных лиц, команда разработки, инструменты для управления требованиями). 

# Оценка по частям (декомпозиция)

Оценка задачи = сумма оценок ее частей.

Маленькие части оценивать легче, чем большие.

# Оценка по трем точкам (Pert)

Оценка = (оптимистическая + пессимистическая + (4 х реалистическая))/6.

# Экспертная оценка

Делегирование права определить оценку человеку, который это сделает лучше вас или вместе с вами.

# Метод Дельфи

1. Эксперты независимо друг от друга оценивают задачи;
2. Их оценки анализируются: если все эксперты дали одинаковый или незначительно отличающиеся оценки, мы получили согласованный результат. 
3. В противном случае, эксперты получают информацию, какие есть альтернативные мнения и могут обосновать свою точку зрения. 
4. Проводиться очередной раунд независимой оценки.

# Метод набегающей волны (Rolling Wave)

1. Грубая предварительная оценка всего задания;
2. Декомпозиция на задачи;
3. Оценка первой задачи;
4. Выполнение первой задачи;
5. Пересмотр оценки всего задания;
6. Переход к следующей задаче.
7. 

Первоначальная оценка может поменяться значительно. Зато вы не откладываете печальную новость, что в срок все готово не будет, на финал.

# Числа Фибоначчи

- Баллы ставятся по экспоненте, чем труднее задача, тем больший промежуток между соседними оценками. 
- Используются целые числа, которые дают более конкретное представление о величине.
- Заставляют сделать выбор между «больше или меньше».
- Убирает заблуждение, что время на задачу можно уменьшить пропорционально, если работать вдвоём, втроём и так далее. Такое заблуждение часто бывает при шкале оценок 2^n (1, 2, 4...).

# Типичные ошибки

Story Point не надо сводить только к сложности, неопределенности или объему работы.

Story Point не переводится в часы.

Story Point не усредняются (3?5 =&gt; 4).

Оценка не меняется в ходе спринта.

Баги нужно оценивать.

Неверно выбранный эквивалентный Story Point =&gt; 0,5 SP

Не подстраивайте оценку в Story Point под отдельного разработчика.

Если меняется состав команды, нужно настроить оценки заново.

Не забывайте обсудить неадекватные оценки на ретроспективе.

# Planning 

Sprint Planning &amp; Planning Poker

# Sprint Planning

— это ограниченная по времени встреча в начале спринта, на которой Команда и Владелец Продукта (PO) обсуждают и принимают решение о том, какая работа будет завершена в спринте.

# Velocity VS Capacity

Velocity

усредненные по времени результаты, полученные командой.

Capacity

это оценочное значение, которое потенциально может быть доставлено, которое определяется во время планирования. Он основан на скорости и адаптирован к реалиям ожидаемых праздников, отсутствия персонала или изменений.

# Planning Poker / Scrum poker

 — техника оценки, основанная на достижении договоренности, главным образом используемая для оценки сложности предстоящей работы или относительного объема решаемых задач при разработке программного обеспечения.

# Проблемы Sprint Planning

Владелец продукта сам определяет и решает, какая работа будет завершена;

Нет доверия или уважения с обеих сторон;

Неподходящая среда, команда не чувствует поддержки или безопасности;

Команда не понимает, для чего нужна эта встреча;

Неактивность команды;

Встреча не включает участников в процесс;

Беклог продукта не актуален, не приоритезирован или не готов к обсуждению;

Излишняя детализация (вся работа уже распределена по исполнителям);

Встреча слишком длинная.

# Stand-up /  Daily sync-up

Stand up be quick

# Stand-up / Daily sync-up

— ежедневное короткое собрание всей команды, целью которой явдяется синхронизация статуса всех членов команды. Заявление статуса происходит по следующему шаблону:

Вчера я занимался вот этим...

Сегодня планирую заниматься вот этим...

У меня есть вот такие блокеры / проблемы...

# Stand-up / Daily sync-up: Цель

подготовка к рабочему дню и планирование его;

оценка предыдущего своего рабочего дня;

поделиться информацией и планами с коллегами;

получить информацию от коллег, которая может пригодится в течение рабочего дня.

# Stand-up / Daily sync-up: Условия

Каждый день в одно и то же время;

Не более 20 минут;

Информация только в рамках шаблона статуса;

Говорит только один человек и в свою очередь.

# Grooming / Refinement

# Grooming (Refinement)

— это регулярное мероприятие, в рамках которого Product Owner совместно с командой проводят анализ и реструктуризацию бэклога. 

Цель — гарантировать, что бэклог заполнен самыми нужными элементами, в должной степени декомпозированными и приоритизированными в соответствии с текущим видением продукта. А также подготовить почву для следующего спринта: проверить, что есть все данные для работы над элементами, которые войдут в будущий спринт. 

# Grooming: Цели

Подготовить задачи в бэклоге для последующей работы с ними: декомпозировать на пользовательские истории, оценить и определить приоритет;

Уточнить актуальность задач, представленных в бэклоге с точки зрения развития продукта;

Прояснить имеющиеся вопросы, получить дополнительную необходимую информацию по задачам, которые пока непонятны и поэтому не могут быть приняты в работу.

# Grooming: Этапы

1. Удаление существующих задач;
2. Добавление новых задач;
3. Декомпозиция задач;
4. Приоритезация задач;
5. Оценка задач;
6. Применение результатов / уроков предыдущих итераций разработки.
