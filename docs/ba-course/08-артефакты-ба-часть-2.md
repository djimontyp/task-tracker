# Преимущества и недостатки SRS

Преимущества

Покрывает как функциональные, так и нефункциональные требования;

Снижает риск недопонимания на ранних стадиях;

Снижает риск утери требований;

Облегчает тестирование;

Помогает запланировать загрузку команды и выявить зависимости.

Недостатки

Подготовка занимает много времени;

Сложна к прочтению и пониманию;

Сложно поддерживать в актуальном состоянии.

# Use Case

Сценарий использования

# Сценарий использования

— это описание поведения системы, когда она взаимодействует с кем-то (или чем-то) из внешней среды.

# Атрибуты

ID + Name;

Description;

Actors;

Pre-conditions;

Trigger;

Main Success Scenario;

Extensions;

Post-condition.

# Пример

| ID+Name               | #123 Login with email & password                                                                                                                                                                                 |
|-----------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Description           | User logs into one’s account                                                                                                                                                                                     |
| Actors                | Registered User                                                                                                                                                                                                  |
| Pre-conditions        | User is not logged in                                                                                                                                                                                            |
| Trigger               | User goes to ‘Login’ page                                                                                                                                                                                        |
| Main success scenario | 1. User fills ‘Email’ field with a email, that one has signed up with 2. User fills ‘Password’ field with a password, that one has signed up with 3. ‘Login’ button becomes active 4. User clicks ‘Login’ button |
| Alternative scenarios | 1a. User enters wrong email 2a. User fills ‘Password’ field  3a. ‘Login’ button becomes active 4a. User clicks ‘Login’ button  5a. ‘There’s no account with such email’ warning message is displayed             |
| Alternative scenarios | 2b. User enters wrong password  3b. ‘Login’ button becomes active 4b. User clicks ‘Login’ button 5b. ‘Wrong password’ warning message is displayed                                                               |
| Post-conditions       | User is logged in and can use one’s account                                                                                                                                                                      |

# Преимущества и недостатки Use Case

Преимущества

Хорошо подходит для оценки, планирования и валидации усилий команды разработки;

Может обновляться в любой итерации;

Альтернативные сценарии покрывают дополнительное поведение;

Понятны и для бизнеса и для разработчиков.

Недостатки

Не объясняет полезности функционала;

Не покрывает нефункциональные требования;

Функционал подается раздельно, не прослеживает зависимость между его частями;

Подготовка занимает немало времени.

# Пример:

User Story #123 Sign in

As a not registered user 

I want sign into system

So that I can create my profile and purchase goods

Acceptance criteria:

user shall have an option to call login form (‘Login’ button in header for guest)

login form shall contain fields for email &amp; password and ‘Submit’ button

‘Email’ field validations rules: shall contain ‘@’ and at least 2 characters after it

‘Password’ field validations rules: min 4, max 8 characters, Latin letters and digits only

user should be notified if the field input doesn’t satisfy validation rules

‘Submit’ button shall be disabled until both fields are filled correctly

# Преимущества и недостатки User Story

Преимущества

Хорошо подходит для оценки, планирования и валидации усилий команды разработки;

Понятны и для бизнеса и для разработчиков;

Быстрая и простая в подготовке;

Достаточно маленькая и понятная для точной оценки;

Ориентирована на пользователя и его потребности;

Отображает полную картину продукта;

Легко описывает разрастание функционала;

Недостатки

Сложно адаптируется для нефункциональных требований;

Не подходит для описания технических задач.

# Когда применять

SRS

Длительные проекты, когда есть достаточно времени для подготовки документации и внешние условия достаточно стабильны

Use Case

Системы, которые содержат множество альтернативных сценариев и требуют соблюдения четкой последовательности действий

User Story

Системы, которые должны быть гибкими и быстро реагировать на внешние изменения. Особенно успешны, при наличии ответственного и вовлеченного Владельца Продукта
