# Специфікація фічі: Executive Summary Epic

**Гілка**: `002-executive-summary`
**Створено**: 2025-12-13
**Статус**: Draft
**Джерело**: PRD-FULL-VISION.md - Epic 2: Weekly Summary (US-010, US-011)

## Огляд

Executive Summary Epic надає CEO та PM тижневий підсумок ключових рішень, блокерів та активності по всіх проєктах. Замість читання сотень повідомлень щодня, керівники отримують структурований звіт для прийняття стратегічних рішень.

**Цільова персона**: Олексій (CEO) — 5-15 хв/день в системі, потребує бачити пульс всіх проєктів за 5 хвилин

**Ключова цінність**: CEO може бути в курсі всіх проєктів без занурення в деталі, тижневий підсумок для стратегічного планування

## Користувацькі сценарії та тестування *(обов'язково)*

### User Story 1 — Тижневий підсумок рішень (Пріоритет: P1)

Як CEO, я хочу бачити тижневий підсумок ключових рішень та блокерів, щоб бути в курсі без читання всіх каналів.

**Чому цей пріоритет**: Це основна цінність для CEO — один погляд на найважливіше за тиждень. Без цього CEO мусить вручну переглядати дашборд щодня.

**Незалежний тест**: Можна повністю протестувати, відкривши сторінку Executive Summary та перевіривши, що відображаються DECISION та BLOCKER атоми за останні 7 днів, згруповані по проєктах/топіках.

**Сценарії приймання**:

1. **Дано** CEO відкриває сторінку Executive Summary, **Коли** сторінка завантажується, **Тоді** він бачить підсумок DECISION атомів за останні 7 днів
2. **Дано** є BLOCKER атоми за тиждень, **Коли** CEO переглядає підсумок, **Тоді** блокери виділені візуально (червоний індикатор) та показані першими
3. **Дано** підсумок завантажується, **Коли** дані отримуються, **Тоді** атоми згруповані по проєктах або топіках для легкої навігації
4. **Дано** CEO переглядає рішення, **Коли** він натискає на атом, **Тоді** він бачить деталі включаючи source message та дату

---

### User Story 2 — Експорт звіту (Пріоритет: P2)

Як CEO, я хочу експортувати тижневий підсумок як звіт, щоб поділитися з командою або зберегти для документації.

**Чому цей пріоритет**: Експорт важливий для sharing та archiving, але система надає цінність і без нього (переглядаючи онлайн).

**Незалежний тест**: Можна протестувати, натиснувши кнопку "Експорт" та перевіривши, що генерується файл у читабельному форматі.

**Сценарії приймання**:

1. **Дано** CEO переглядає Executive Summary, **Коли** він натискає "Експорт", **Тоді** система генерує звіт у форматі, придатному для читання (текст, markdown)
2. **Дано** звіт експортовано, **Коли** CEO відкриває файл, **Тоді** він містить всі рішення та блокери з підсумку, структуровані по секціях
3. **Дано** CEO хоче поділитися звітом, **Коли** він копіює вміст, **Тоді** форматування зберігається для вставки в чат або email

---

### User Story 3 — Вибір періоду (Пріоритет: P2)

Як CEO, я хочу обирати період для підсумку (тиждень, 2 тижні, місяць), щоб бачити різні горизонти активності.

**Чому цей пріоритет**: Гнучкість періоду корисна, але стандартний тиждень покриває 90% use cases.

**Незалежний тест**: Можна протестувати, змінивши період у селекторі та перевіривши, що підсумок оновлюється відповідно.

**Сценарії приймання**:

1. **Дано** CEO на сторінці Executive Summary, **Коли** він обирає "Останній тиждень", **Тоді** показуються атоми за останні 7 днів
2. **Дано** CEO обирає "Останні 2 тижні", **Коли** підсумок оновлюється, **Тоді** показуються атоми за останні 14 днів
3. **Дано** CEO обирає "Останній місяць", **Коли** підсумок оновлюється, **Тоді** показуються атоми за останні 30 днів
4. **Дано** період за замовчуванням, **Коли** CEO відкриває сторінку, **Тоді** показується "Останній тиждень" (7 днів)

---

### User Story 4 — Крос-проєктний огляд (Пріоритет: P3)

Як CEO, я хочу бачити активність по всіх проєктах в одному місці, щоб ідентифікувати патерни та порівнювати прогрес.

**Чому цей пріоритет**: Корисно для компаній з кількома проєктами, але можна реалізувати пізніше через фільтрацію.

**Незалежний тест**: Можна протестувати, переглянувши крос-проєктний view з метриками активності по кожному проєкту.

**Сценарії приймання**:

1. **Дано** CEO на сторінці Executive Summary, **Коли** він переглядає крос-проєктний view, **Тоді** він бачить всі проєкти з метриками активності (кількість рішень, блокерів, повідомлень)
2. **Дано** є кілька проєктів, **Коли** CEO порівнює їх, **Тоді** він бачить порівняльну таблицю або картки з ключовими показниками
3. **Дано** CEO хоче деталі проєкту, **Коли** він натискає на проєкт, **Тоді** він переходить до відфільтрованого підсумку тільки для цього проєкту

---

### Граничні випадки

- Що відбувається коли немає атомів за обраний період? Показати порожній стан "Немає активності за цей період" з пропозицією обрати довший період
- Як система обробляє дуже великий підсумок (100+ рішень)? Пагінація або "Показати більше" з початковим лімітом 20 елементів
- Що якщо проєкт не має жодного рішення? Показати проєкт з індикатором "Немає рішень" (не приховувати)
- Як обробляється експорт з великою кількістю даних? Показати прогрес генерації, генерувати у фоновому режимі
- Що якщо є блокери старші 2 тижнів? Виділяти "застарілі" блокери як критичні (потребують уваги)
- Як система поводиться при зміні періоду з незавершеними даними? Показати стан завантаження, не скидати попередні дані до завершення

## Вимоги *(обов'язково)*

### Функціональні вимоги

#### Тижневий підсумок (US-010)
- **FR-001**: Система ПОВИННА відображати окрему сторінку Executive Summary доступну з головної навігації
- **FR-002**: Система ПОВИННА показувати DECISION атоми за обраний період (за замовчуванням 7 днів)
- **FR-003**: Система ПОВИННА показувати BLOCKER атоми, виділені візуально
- **FR-004**: Система ПОВИННА групувати атоми по проєктах або топіках
- **FR-005**: Система ПОВИННА відображати дату та source для кожного атома
- **FR-006**: Система ПОВИННА показувати загальну статистику (кількість рішень, блокерів, проєктів з активністю)

#### Експорт
- **FR-007**: Система ПОВИННА надавати можливість експортувати підсумок у текстовому форматі
- **FR-008**: Експортований звіт ПОВИНЕН містити всі рішення та блокери, структуровані по секціях
- **FR-009**: Система ПОВИННА зберігати форматування при копіюванні (для вставки в чат/email)

#### Вибір періоду
- **FR-010**: Система ПОВИННА надавати селектор періоду (7 днів, 14 днів, 30 днів)
- **FR-011**: Система ПОВИННА зберігати обраний період в сесії користувача
- **FR-012**: Система ПОВИННА оновлювати підсумок при зміні періоду без перезавантаження сторінки

#### Крос-проєктний огляд (US-011)
- **FR-013**: Система ПОВИННА показувати всі проєкти з метриками активності
- **FR-014**: Система ПОВИННА дозволяти фільтрацію підсумку по конкретному проєкту
- **FR-015**: Система ПОВИННА показувати порівняльні метрики між проєктами

### Ключові сутності

- **Executive Summary**: Агрегований звіт за період, містить рішення (DECISION), блокери (BLOCKER), статистику. Генерується динамічно на основі approved атомів.
- **Summary Period**: Часовий діапазон для фільтрації (7, 14, 30 днів), зберігається як user preference для сесії.
- **Project Metrics**: Агреговані показники по проєкту — кількість рішень, блокерів, активних топіків, загальна активність.
- **Export Report**: Текстова версія підсумку для sharing, структурована по секціях (Рішення, Блокери, Статистика).

## Критерії успіху *(обов'язково)*

### Вимірювані результати

- **SC-001**: CEO може отримати повну картину по всіх проєктах за 5 хвилин (вимірюється через час на сторінці Executive Summary)
- **SC-002**: Executive Summary завантажується менш ніж за 3 секунди для періоду 7 днів з 50+ атомами
- **SC-003**: Користувачі переглядають Executive Summary щонайменше раз на тиждень (вимірюється через analytics)
- **SC-004**: 100% BLOCKER атомів видимі та виділені в підсумку (перевірка completeness)
- **SC-005**: Експорт звіту генерується менш ніж за 5 секунд для підсумку з 100 атомами
- **SC-006**: Користувачі можуть поділитися звітом через copy-paste без втрати форматування (перевірка в чаті)
- **SC-007**: Зміна періоду оновлює дані менш ніж за 2 секунди
- **SC-008**: Крос-проєктний view показує метрики для всіх проєктів (100% coverage)

## Припущення

- Атоми (DECISION, BLOCKER) вже створюються AI-екстракцією та затверджуються через Daily Review (001-epic)
- Існує щонайменше один проєкт з топіками для групування атомів
- Кількість атомів за тиждень типово 20-100 (на основі 1000-5000 повідомлень/день)
- Користувачі розуміють типи атомів та їх значення для бізнесу
- Українська мова є основною мовою UI та контенту
- Всі користувачі мають однакові права доступу до Executive Summary (без ролей)
- Експорт в простому текстовому форматі достатній (PDF/Word не потрібен для MVP)

## Залежності

- **001-daily-review-epic**: Атоми повинні існувати та бути затверджені для відображення в підсумку
- **Topics/Projects**: Групування по проєктах вимагає існуючої структури топіків
